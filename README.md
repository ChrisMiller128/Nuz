# Nuzlocke Hub

> Play, track, and conquer Pokémon Nuzlocke runs directly in your browser.

**Live:** [https://nuzlocke.emulator.st](https://nuzlocke.emulator.st)

---

## Architecture Overview

Nuzlocke Hub is a full-stack web application that lets users:

- **Play** Pokémon games (GB, GBC, GBA, NDS) in browser-based emulators
- **Generate** Nuzlocke-ready modified ROMs with customizable randomizer settings
- **Track** encounters, party, deaths, badges, and journal entries per run
- **Save** progress server-side and resume from any device
- **Own** their generated ROMs as persistent, per-user durable assets

## Why This Stack

| Component | Choice | Rationale |
|-----------|--------|-----------|

| ORM | **Prisma** | Type-safe queries, migration management, schema-as-code |
| Database | **PostgreSQL 16** | Robust relational model for complex entity relationships |

| Emulator | **EmulatorJS** | Unified RetroArch WASM wrapper for multiple platforms |
| Containers | **Docker Compose** | Reproducible deployment, isolated services |
| Proxy | **Nginx** | SSL termination, caching, security headers |

## Setup

### Prerequisites

- Debian 13 (or compatible Linux)
- Root/sudo access
- Domain DNS pointing to server (for SSL)

### Quick Deploy

```bash
chmod +x deploy.sh
sudo ./deploy.sh
```

The script will:
1. Install Docker, Nginx, Certbot if missing
2. Build the Next.js application in Docker
3. Start PostgreSQL and the app
4. Run database migrations and seed data
5. Configure Nginx reverse proxy
6. Attempt SSL certificate via Let's Encrypt
7. Print status and next steps

### Manual Setup (Development)

```bash

# Set up environment

# Edit .env with your database URL and secrets

# Generate Prisma client
npx prisma generate

# Push schema to database
npx prisma db push

# Seed the database
npx tsx scripts/seed.ts

# Run development server
npm run dev
```

### Data Models

- **User** — Account with email, username, password hash
- **Game** — Pokémon game entries (Red, Blue, Emerald, etc.)
- **BaseRom** — Platform-managed ROM files per game
- **GeneratedRom** — User-owned generated/modified ROM artifacts
- **GeneratorProfile** — Reusable generator presets
- **Run** — Nuzlocke run with rules, status, and tracker data
- **SaveState** — Server-side save files per run/slot
- **Encounter** — Route-based encounter tracking
- **PokemonEntry** — Party/box/graveyard Pokémon with full metadata
- **BadgeProgress** — Badge and level cap tracking
- **RunNote** — Journal entries per run
- **DeviceSession** — Cross-device session tracking

## Save System Design

Saves are stored server-side and linked to specific runs and generated ROMs:

1. **Autosave** — The emulator periodically captures state and uploads via `/api/saves`
2. **Manual Save** — Users can trigger saves from the emulator control bar
3. **Slot-based** — Each run has numbered save slots (0 = autosave by default)
4. **Cross-device** — Saves are fetched from server when resuming on any device
5. **Conflict model** — Latest-autosave-wins (timestamp-based)
6. **Download** — Users can download save files for backup

### Save Data Flow

```
Browser Emulator → ArrayBuffer → FormData POST /api/saves
                                      ↓
                              File saved to disk
                              SaveState DB record upserted
                                      ↓
                              Run.lastPlayedAt updated
```

## Emulator Abstraction Design

The emulator layer uses an adapter pattern to support multiple platforms:

```
EmulatorWrapper (React Component)
    ├── Loads EmulatorJS from CDN
    ├── Configures core based on platform:
    │   ├── GB/GBC → gambatte core
    │   ├── GBA    → mgba core
    │   └── NDS    → melonds core
    ├── Handles save state capture/restore
    ├── Provides control bar (pause/resume/save/fullscreen)
    └── Desktop keyboard + mobile touch controls
```

EmulatorJS wraps RetroArch cores compiled to WebAssembly, providing broad compatibility across platforms.

## Generator Architecture

The ROM generator service is a modular abstraction that produces real generated ROM artifacts:

### Flow

1. User selects base game/ROM + settings + optional seed
2. `generateRom()` is called with settings and base ROM reference
3. A deterministic PRNG seeded from the user's seed applies modifications
4. The modified ROM is written to persistent storage under the user's directory
5. A `GeneratedRom` database record is created with full metadata
6. The run references this `GeneratedRom` record
7. On future sessions, the **same saved ROM** is loaded (no regeneration)

### Generated ROM Ownership Model

Generated ROMs are **first-class durable assets**, not temporary files:

- Each GeneratedRom belongs to exactly one User
- Each GeneratedRom references its source BaseRom
- Each GeneratedRom stores the full generator settings JSON and seed
- Each GeneratedRom has a SHA-256 checksum for integrity verification
- The ROM file is stored on disk under the user's storage directory
- The database record persists the storage path, file name, and file size
- Runs always reference their GeneratedRom without regenerating
- Users can view, download, and manage their generated ROMs

### Extension Points

The generator service architecture is designed for future enhancement:

- **Real randomizer integration**: Replace `applyGeneratorModifications()` with a call to Universal Pokemon Randomizer ZX (Java subprocess) or a WASM-compiled randomizer
- **Per-game handlers**: Add platform-specific ROM patching logic
- **Preset library**: Use `GeneratorProfile` records for shareable presets
- **Seed sharing**: Users can share seeds to reproduce exact ROM modifications

## ROM Legality Note

**Nuzlocke Hub does not include or distribute any Pokémon ROM files.** Users must provide their own legally obtained ROM files. The application only provides:

- A game catalog (title, platform, metadata)
- Placeholder base ROM entries for the generator workflow
- Tools to modify and manage user-provided ROM data

Please ensure you own legal copies of any games you use with this platform.
